# Effective Java 實戰筆記 (三)：類別與介面的設計

類別與介面是 Java 系統架構的基石。在這個章節中，Joshua Bloch 傳授了許多關於封裝、繼承與介面設計的黃金守則。你會發現，許多我們初學時習以為常的物件導向習慣（例如無腦使用繼承），其實在大型專案中可能會成為災難。

這篇文章整理了《Effective Java》第三版中關於類別與介面的核心精華，並附上具體的實戰範例。

## 1. 最小化類別與成員的存取權限

封裝 (Encapsulation) 是軟體設計的靈魂。一個設計良好的模組會把實作細節隱藏起來，只對外暴露必要的 API，降低系統各模組之間的耦合度。

**原則：**

- 盡可能讓每個類別或成員的存取層級降到最低（預設先用 `private`，有需要再開放為 `package-private`）。
- **特別注意：長度不為零的陣列總是可變的！** 就算你把它宣告為 `public static final`，外部依然可以修改裡面的元素。

❌ **錯誤示範：暴露內部陣列結構**

```java
// 潛在的安全漏洞：外部可以輕易修改 BAND_MEMBERS[0] = "Other Band";
public static final String[] BAND_MEMBERS = { "Casiopea", "T-Square" };
```

✅ **正確寫法：使用不可變的 List 或回傳 Clone**

```java
private static final String[] BAND_MEMBERS = { "Casiopea", "T-Square" };

// 做法 A：暴露不可變的 List
public static final List<String> BAND_MEMBERS_LIST =
    Collections.unmodifiableList(Arrays.asList(BAND_MEMBERS));

// 做法 B：回傳陣列的拷貝
public static final String[] values() {
    return BAND_MEMBERS.clone();
}
```

## 2. 在公開類別中，使用存取子方法而非公開欄位

如果一個類別是可以被套件外部存取的（Public Class），它的實體欄位 (Instance fields) 永遠不該被設為 public！如果直接公開欄位，你將會失去封裝的所有好處（例如無法在賦值時進行邏輯驗證，也無法改變內部實作）。

❌ **錯誤示範：暴露內部欄位**

```java
public class WaferLot {
    // 絕對不要這樣做！外部可以隨意竄改甚至引發 Thread-safe 問題
    public String lotId;
    public int yieldRate;
}
```

✅ **正確寫法：使用私有欄位與存取子 (Getter/Setter)**

```java
public class WaferLot {
    private String lotId;
    private int yieldRate;

    public WaferLot(String lotId, int yieldRate) {
        this.lotId = lotId;
        this.yieldRate = yieldRate;
    }

    public String getLotId() { return lotId; }

    // 可以在 setter 中加入邏輯驗證，確保內部狀態的一致性
    public void setYieldRate(int yieldRate) {
        if (yieldRate < 0 || yieldRate > 100) throw new IllegalArgumentException();
        this.yieldRate = yieldRate;
    }
}
```

## 3. 盡可能讓類別不可變

不可變物件 (Immutable Object) 是指在建立之後，狀態就永遠不會改變的物件。它們本質上就是**執行緒安全 (Thread-safe)** 的，不需要同步鎖定就能讓多個執行緒自由共享。

**如何打造不可變類別？**

1. 不提供任何修改狀態的方法 (No mutators/setters)。
2. 保證類別不能被擴充 (例如聲明為 `final`)。
3. 所有的欄位都聲明為 `final` 與 `private`。

**實戰範例：**
假設我們撰寫一套 Playwright 自動化腳本來買 Billboard Live 的演唱會門票，每一張搶到的票都應該是不可變的：

```java
// 加上 final 避免被子類別繼承破壞
public final class ConcertTicket {
    private final String bandName;
    private final String seat;

    public ConcertTicket(String bandName, String seat) {
        this.bandName = bandName;
        this.seat = seat;
    }

    public String getBandName() { return bandName; }
    public String getSeat() { return seat; }

    // 如果需要「修改」，則是回傳一個「全新」的實例，而不是變動原本的物件
    public ConcertTicket upgradeSeat(String newSeat) {
        return new ConcertTicket(this.bandName, newSeat);
    }
}
```

> 💡 **Tip:** 除非有強烈的效能考量或框架要求，否則所有的類別都應該盡可能設計成不可變的！

## 4. 多用合成 (Composition)，少用繼承

在物件導向的世界裡，繼承 (`extends`) 雖然是個強大的武器，但它卻會**嚴重破壞封裝性**。

當你繼承一個具體類別時，子類別會變得極度依賴父類別的「內部實作細節」。只要父類別在未來的版本中稍微修改了底層邏輯，子類別可能就會在毫無預警的情況下崩潰。

### ❌ 繼承的恐怖踩坑現場：被重複計算的集合

假設我們想自訂一個 `HashSet`，用來記錄「總共被加入過多少張照片」：

```java
public class InstrumentedPhotoSet<E> extends HashSet<E> {
    private int addCount = 0;

    @Override
    public boolean add(E e) {
        addCount++;
        return super.add(e);
    }

    @Override
    public boolean addAll(Collection<? extends E> c) {
        addCount += c.size();
        return super.addAll(c); // 這裡藏著致命危機！
    }
    public int getAddCount() { return addCount; }
}
```

如果我們一次呼叫 `addAll()` 加入 3 張照片，你以為 `addCount` 會是 3，但結果居然是 6！因為 Java 底層的 `HashSet.addAll()` 其實是跑迴圈呼叫 `add()`。當你呼叫 `super.addAll()` 時，它又觸發了你覆寫的 `add()`，導致重複計算。

### ✅ 救星來了：合成 (Composition) 與轉發 (Forwarding)

與其直接 `extends`，不如在類別內部「包含 (compose)」一個現有類別的實例，並將所有的操作「轉發 (forward)」給它。

```java
// 實作 Set 介面，並在內部「合成」一個 Set
public class InstrumentedPhotoSet<E> implements Set<E> {
    private int addCount = 0;
    private final Set<E> s; // 把別人當成自己的一個小零件

    public InstrumentedPhotoSet(Set<E> s) { this.s = s; }

    @Override
    public boolean add(E e) {
        addCount++;
        return s.add(e);
    }

    @Override
    public boolean addAll(Collection<? extends E> c) {
        addCount += c.size();
        return s.addAll(c); // 這裡不再會有重複計算的問題！
    }

    public int getAddCount() { return addCount; }
    // ... 其餘 Set 介面的方法，全部單純轉發給 s
}
```

## 5. 設計可繼承的類別，否則就禁止繼承

如果你決定不聽上一條的勸告，堅持要開放類別讓別人繼承，那麼你必須負起極大的責任：**你要公開並文件化類別的內部行為細節**。

**地雷一：自我使用 (Self-use) 未標示清楚**
如果你的類別內部方法會互相呼叫，且這些方法可以被覆寫，你必須在文件中（例如使用 `@implSpec` 標籤）寫清楚呼叫順序與邏輯。

**地雷二：在建構子裡呼叫可覆寫的方法**
這是一個極度常見且致命的錯誤。**建構子絕對不能呼叫可被覆寫的方法！**

❌ **致命的踩坑現場：未初始化的 Bass 效果器**

```java
public class AudioEffectPedal {
    public AudioEffectPedal() {
        // 絕對不要在建構子呼叫可覆寫的方法！
        applyEffect();
    }
    public void applyEffect() { }
}

public class BassFuzzPedal extends AudioEffectPedal {
    private final String fuzzType;

    public BassFuzzPedal() {
        // 父類別的建構子會先執行，此時這行根本還沒跑！
        fuzzType = "Vintage Germanium";
    }

    @Override
    public void applyEffect() {
        // 第一次被父類別建構子呼叫時，fuzzType 會是 null！
        System.out.println("Applying Fuzz: " + fuzzType.toUpperCase());
    }
}
```

當你 `new BassFuzzPedal()` 時，預期會印出效果名稱，但結果卻是直接噴出 `NullPointerException`！

> 💡 **防禦法則：** 最保險的做法是「沒準備好，就別讓人繼承你的類別！」直接把類別宣告為 `final`，或者把建構子設為私有並提供靜態工廠方法。

## 6. 偏好介面而非抽象類別

Java 只允許單一繼承，但可以實作多個介面。這讓介面成為定義「混合功能 (Mixins)」的完美選擇。現有類別也很容易藉由實作新介面來擴充功能。

例如，我們在建立一個音樂人的類別時，介面能賦予極大的彈性：

```java
public interface BassPlayer { void slap(); }
public interface Songwriter { void compose(); }

// 透過介面，我們可以輕鬆組合不同的能力，而不會受限於單一繼承樹
public class JazzFusionMusician implements BassPlayer, Songwriter {
    @Override
    public void slap() { System.out.println("Slapping the bass!"); }

    @Override
    public void compose() { System.out.println("Writing a new track."); }
}
```

## 7. 謹慎設計介面，預防 Default Method 的陷阱

自從 Java 8 引入了 `default method`（預設方法）後，我們終於可以在不破壞現有實作的情況下為介面擴充新功能。但這潛藏著極大的相容性風險。

當你在介面中新增一個 `default method` 時，舊有的實作類別根本不知道這個新方法的存在。
例如 Java 8 在 `Collection` 介面新增了 `removeIf()` 預設方法，但 Apache Commons 的 `SynchronizedCollection` 卻沒有覆寫它。結果開發者呼叫 `removeIf()` 時，直接繼承了**沒有任何同步保護**的預設實作，導致在多執行緒下引發 `ConcurrentModificationException` 或資料錯亂。

> 💡 **防禦法則：** 新增 `default method` 就像在運作中的系統動手術。在設計新介面時，請依然保持嚴謹，不要把 `default method` 當成事後彌補設計缺陷的藉口。

## 8. 介面只用來定義型別，拒絕「常數介面」反模式

介面的本質是用來定義「這個物件可以做什麼行為」。把介面當成了「常數倉庫」，被稱為**常數介面反模式 (Constant Interface Antipattern)**。

❌ **錯誤示範：常數介面反模式**

```java
public interface PhysicalConstants {
    static final double ELECTRON_MASS = 9.109_383_56e-31;
}
```

當別的類別為了貪圖方便去 `implements` 它時，這些僅供內部使用的常數就會被迫暴露到該類別的 API 中，嚴重污染了類別設計。

✅ **正確寫法：使用不可實例化的工具類別 (Utility Class)**

```java
public class PhysicalConstants {
    private PhysicalConstants() {} // 阻止實例化
    public static final double ELECTRON_MASS = 9.109_383_56e-31;
}
```

如果不想一直打類別名稱，請善用 `import static`。

## 9. 類別階層優於「標籤類別」

有時候我們會看到一個類別裡面塞滿了 `enum` 標籤跟 `switch` 語句。這種「標籤類別」非常冗長、浪費記憶體，且極易出錯。

❌ **錯誤示範：標籤類別**

```java
class CameraLens {
    enum Type { PRIME, ZOOM };
    final Type type;
    double focalLength; // 只有定焦鏡才需要
    double minFocalLength; // 只有變焦鏡才需要
    // 一大堆建構子跟 switch 語句...
}
```

✅ **正確寫法：類別階層 (Class Hierarchy)**

```java
abstract class CameraLens { abstract void mount(); }

class PrimeLens extends CameraLens {
    final double focalLength;
    PrimeLens(double focalLength) { this.focalLength = focalLength; }
    void mount() { /* 掛載定焦鏡的邏輯 */ }
}

class ZoomLens extends CameraLens {
    final double minFocalLength, maxFocalLength;
    ZoomLens(double min, double max) { ... }
    void mount() { /* 掛載變焦鏡的邏輯 */ }
}
```

## 10. 優先考慮靜態成員類別

如果你需要在類別內部定義一個巢狀類別（Nested Class），請牢記一個大原則：**如果它不需要存取外部類別的實例，請務必加上 `static` 宣告！**

非靜態的內部類別會在底層隱藏一個指向外部類別的參考 (Reference)。這不但浪費記憶體，還會導致外部物件無法被垃圾回收 (GC)，成為潛在的 Memory Leak 殺手。

```java
public class Calculator {
    // 沒有需要用到 Calculator 實體的變數或方法，果斷加上 static！
    public static enum Operation { PLUS, MINUS, MULTIPLY, DIVIDE }
}
```

## 11. 將原始碼檔案限制為單一頂層類別

Java 編譯器其實允許你在同一個 `.java` 檔案中定義多個頂層類別 (Top-level classes)，但**這是一個絕對不該踩的超級地雷**。

如果你在 `Main.java` 裡面同時定義了 `class A` 跟 `class B`，而專案中剛好又有其他的檔案也定義了 `A` 或 `B`。編譯器並不會立刻報錯，而是會**根據編譯器吃檔案的順序，決定覆蓋哪一個實作**！這種在編譯時期偷偷切換行為的 Bug 幾乎無法靠肉眼除錯。

✅ **唯一正解：** 永遠保持「一個 `.java` 檔案，只裝一個頂層類別」。如果兩個類別真的高度關聯，請使用上一點提到的**靜態成員類別 (Static Member Class)** 將它們包在一起。
