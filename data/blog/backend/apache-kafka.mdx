## 1. 核心資料結構：Topics, Partitions 與 Offsets

![Kafka Core Architecture](/static/images/blog/kafka/kafka-core-architecture.png)

在 Kafka 中，資料的儲存與分類依賴於 Topic 與 Partition 的設計。每一個 Topic 可以由多個 Partition 組成，這賦予了 Kafka 極高的吞吐量與擴展性。

- **不可變性 (Immutability)**：資料一旦被寫入 Partition 中，就無法再被更改。
- **資料留存 (Retention)**：Kafka 預設會將資料保存一段時間（預設為一週），過期後自動清除。
- **Offset 的意義**：Offset 只是一個存在於特定 Partition 中的流水號，用來標示訊息的位置。
- **順序性保證**：Kafka **只有在同一個 Partition 內** 才能保證訊息的順序性。跨 Partition 的訊息是沒有絕對順序的。
- **資料分發**：預設情況下，資料會被隨機（Round-robin）送到不同的 Partition 中，除非你在寫入時有特別指定 Key。

## 2. 訊息生產者：Producers

Producer 負責將資料寫入 Topic 中。它非常聰明，能夠自動知道應該將資料寫往哪一個 Partition，且萬一遇到 Broker 故障，Producer 也具備自動復原與重試的機制。

### Message Keys 與路由機制

Producer 在發送訊息（Message）時，可以選擇性地帶上一個 Key：

- **`Key = null`**：訊息會以 Round-robin 的方式（0, 1, 2...）均勻分配到各個 Partition。
- **`Key != null`**：透過雜湊演算法，**擁有相同 Key 的所有訊息將始終被發送到同一個 Partition**。

![Message Keys](/static/images/blog/kafka/message-keys.png)

> [!NOTE]
> 實務應用場景：如果你需要針對特定欄位（例如：同一張訂單 `order_id`）的訊息保證處理順序，通常會將該欄位設為 Key。

### 序列化 (Serializer)

Kafka 的底層傳輸只接受 Bytes 格式的 Input 與 Output。因此，Producer 建立的 Message（包含 Key 與 Value）在送出前，都必須先通過 **Serializer** 將物件或資料結構轉換成 Bytes。

![Serializer](https://hackmd.io/_uploads/BkBvhF_sp.png)

## 3. 訊息消費者：Consumers

Consumer 負責從 Topic 中讀取資料。它同樣知道該連線到哪一個 Broker，也能在 Broker 發生異常時自動切換。讀取時，會依照 Offset 從低到高的順序進行（順序僅限單一 Partition）。

![Consumer](https://hackmd.io/_uploads/SkOk6F_oT.png)

### 反序列化 (Deserializer)

與 Producer 相對應，Consumer 收到 Bytes 後，需要透過 Deserializer 將其還原為原本的物件或資料格式。

- 注意：Topic 的資料型態在生命週期內不應該被改變，否則會導致反序列化失敗。

### Consumer Groups (消費者群組)

在實際的微服務應用中，Consumer 通常會以群組（Group）的形式運作，透過 `group.id` 來區分：

- 每一個 Consumer Group 內部的 Consumer 會被分配去讀取特定的 Partition，確保同一筆訊息在同一個群組內只會被處理一次。
- 同一個 Topic 可以同時被「多個不同的 Consumer Groups」訂閱與讀取，彼此互不影響。

![Consumer Groups 1](https://hackmd.io/_uploads/SyaMRY_iT.png)
![Consumer Groups 2](https://hackmd.io/_uploads/BJpwkqOop.png)

### Consumer Offsets

Kafka 需要記住每個 Consumer Group 讀到哪裡了：

- Broker 會週期性地將 Consumer 提交 (Commit) 的 Offset 儲存在一個特殊的 Topic 中（`__consumer_offsets`）。
- 如果某個 Consumer 突然 Crash，當它重啟或有其他 Consumer 接手時，就會去尋找最後一次 Commit 的 Offset，從中斷的地方繼續處理。

![Consumer Offsets](https://hackmd.io/_uploads/HkzTeqOsp.png)

### 傳遞語意 (Delivery Semantics)

- **At least once (至少一次)**：Java Client 的預設與首選。在處理完訊息後才 Commit。若處理過程發生錯誤或重啟，訊息會被再次讀取（可能導致重複處理，需搭配冪等性設計）。
- **At most once (至多一次)**：收到訊息當下立刻 Commit。若後續處理失敗，該訊息就會永久遺失。
- **Exactly once (精確一次)**：保證只處理一次。通常需要結合 Kafka 的 Transactional API 或外部系統的冪等性（Idempotent）消費者來實現。

## 4. 叢集架構與高可用性：Brokers & Zookeeper

### Brokers

Kafka 叢集由多個 Broker 伺服器組成，每個 Broker 都有獨立的 ID，並負責管理特定的 Topic Partitions。

- **Bootstrap Server**：你只需要連上叢集中的任意一個 Broker，Kafka Client 就會自動獲取整個叢集的 Metadata 並連上其他節點。
- 實務建議：一個穩健的叢集通常至少需要 3 個以上的 Brokers。

![Brokers](https://hackmd.io/_uploads/H1cJX9doT.png)

### Topic Replication (副本機制)

為了防止資料遺失，**Replication Factor 必須大於 1**。這讓 Kafka 容許在 N-1 個 Broker 故障的情況下，依然能提供服務並不掉資料。

![Replication](https://hackmd.io/_uploads/r1nEXcui6.png)

#### Partition Leader 概念

- 在任何時間點，一個 Partition 只會有一個 Broker 擔任 **Leader**。
- 預設情況下，Producers 與 Consumers **只能對該 Partition 的 Leader 進行讀寫**。
- 其他擁有該 Partition 副本的 Broker 稱為 Follower（處於 ISR - In-Sync Replicas 清單中），它們會在背景被動地從 Leader 抄寫資料。

> [!NOTE]
> 進階優化：自 Kafka 2.4 版本起，可以設定讓 Consumer 從地理位置最近的 Replica 讀取資料，藉此改善網路延遲 (Latency)。

### Producer Acknowledgements (acks)

Producer 可以設定發送訊息後等待 Broker 回應的策略：

- `acks=0`：射後不理，不等待 ACK（可能遺失資料，但延遲最低）。
- `acks=1`：只等待 Leader 寫入成功的 ACK（有限的資料遺失風險）。
- `acks=all` (-1)：等待 Leader 加上所有 ISR 副本都寫入成功的 ACK（無資料遺失，最安全但延遲較高）。

![Acks](https://hackmd.io/_uploads/SycAVqusa.png)

### Zookeeper 與 KRaft

在傳統架構中，Kafka 高度依賴 Zookeeper：

- 負責管理 Brokers 清單。
- 負責在 Partitions 中選出 Leader 並發送通知給 Kafka。
- **注意**：Zookeeper 不儲存 Consumer Offsets，這是 Kafka 自己的工作。
- 伺服器數量必須是奇數（1, 3, 5, 7），由一個 Leader 負責寫入，其餘 Followers 負責讀取。

> [!NOTE]
> 架構演進：根據 KIP-500，自 Kafka 3.X 版本之後，Kafka 已逐步引進 KRaft (Kafka Raft) 共識機制，可以完全移除對 Zookeeper 的依賴。

## 5. Kafka CLI 實戰指令

### 環境安裝 (Mac)

```bash
brew install kafka

# 啟動 Zookeeper
/usr/local/bin/zookeeper-server-start /usr/local/etc/zookeeper/zoo.cfg

# 啟動 Kafka Broker
/usr/local/bin/kafka-server-start /usr/local/etc/kafka/server.properties
```

### Topic CLI 管理操作

```bash
# 建立簡單的 Topic
kafka-topics --bootstrap-server localhost:9092 --topic first_topic --create

# 建立指定 Partition 數量的 Topic
kafka-topics --bootstrap-server localhost:9092 --topic second_topic --create --partitions 3

# 建立包含 Partition 與 Replication Factor 的 Topic
# (注意：Replication 數量不能大於叢集中的 Broker 總數)
kafka-topics --bootstrap-server localhost:9092 --topic third_topic --create --partitions 3 --replication-factor 1

# 列出所有 Topics
kafka-topics --bootstrap-server localhost:9092 --list

# 查看特定 Topic 的詳細資訊 (Leader, Replicas, ISR)
kafka-topics --bootstrap-server localhost:9092 --describe

# 刪除 Topic
kafka-topics --bootstrap-server localhost:9092 --topic second_topic --delete
```
