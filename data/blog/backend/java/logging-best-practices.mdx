**目標讀者**：Spring Boot 後端工程師，負責設計或審查日誌策略
**先備知識**：了解 `logback`/`slf4j` 基本用法，知道 DEBUG/INFO/WARN/ERROR 四個等級的存在
**學習成果**：能立即套用正確的 Log 分級策略，避免在 Production 噴出大量雜訊或敏感資料

---

日誌（Logging）真的是區分菜鳥跟老鳥的一大指標。好的 Log 讓你在半夜被叫醒時，三分鐘抓出兇手；壞的 Log 只會讓你在幾百萬行雜訊裡懷疑人生。

## 1. 嚴格遵守 Log 分級

- **DEBUG：** 開發/測試環境專用。紀錄 SQL、輸入參數、中間計算結果。正式環境絕對關閉。
- **INFO：** 正式環境保留。紀錄系統健康狀態（服務啟動、Job 完成、呼叫重要外部 API），別在迴圈裡狂噴 INFO。
- **WARN：** 潛在風險。系統異常但有替代方案（如用預設值替代缺失配置、API 延遲接近閾值），需關注但不需半夜起床。
- **ERROR：** 必須告警。資料庫連線失敗、交易中斷。出現 ERROR 就是要修，通常綁定 Alert 發送通知。

## 2. 實務踩坑技巧

- **精簡內容：** 禁止直接把整個大物件印出來 `log.info("user: {}", user)`。只要紀錄關鍵屬性與 UUID，避免效能問題跟敏感資料外洩。
- **結構化日誌 (Structured Logging)：** 為了讓 ELK 好搜，盡量用 JSON 格式或具名標籤（例如 `orderId=123 | userId=456`）。
- **上下文資訊：** 紀錄 HTTP 來源 IP、處理時間，微服務記得用 MDC 塞入 `traceId` 串起整個 Request。**嚴格屏蔽密碼與個資**。
